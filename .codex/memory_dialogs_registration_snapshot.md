- Лента `/feed` уже поддерживает фильтры по источникам, сортировку и медиаблоки.

## 2) Оценка готовности по условию «медиа в ленте полностью готовы»
Текущий статус: **частично готово**.

Почему не «полностью»:
1. В TODO остается задача о переходе с placeholder URL на реальные публичные ссылки из Blob/S3.
2. В TODO остается задача о повторном TG-импорте после докидки исходных медиа-файлов.

Вывод: отдельный snapshot под «диалоги + регистрация» логично делать после закрытия двух задач выше, иначе в новой ветке придется дублировать миграцию медиа-ссылок.

## 3) Что уже можно подготовить сейчас (без блокеров по медиаконтенту)
### Диалоги
- Зафиксировать MVP-контракт диалогов (модель, поля, пагинация, read-state) в отдельном RFC/ADR.
- Добавить минимальные миграции под сущности диалогов и сообщений (без разрыва текущего API).
- Заложить API-скелет: `/api/dialogs`, `/api/dialogs/{id}/messages` + rate-limit/authorization hooks.

### Регистрация
- Подготовить MVP flow: регистрация по email + верификация через SMTP.
- Выделить env-профиль (`env/auth.env.example`) и безопасные defaults.
- Описать инварианты: уникальность email, токен верификации с TTL, idempotent resend.

## 4) Предлагаемый момент для snapshot-ветки
Создавать ветку вида `snapshot/2026-03-dialogs-registration` после:
- закрытия TODO по реальным media URL;
- повторного TG-импорта с полным медиа-набором;
- sanity-проверки `/feed` на 2-3 страницах и разных source-фильтрах.

1. Закрыть media-blockers (storage + backfill URL + повторный импорт TG).
2. Сформировать технический план по диалогам/регистрации в виде задач P1/P2.
3. После закрытия blockers — открыть snapshot-ветку и вести там изолированные изменения auth+dialogs.
