## Обзор

### 1. Слабые стороны

*   **Масштабируемость**
    *   **Серьезность**: СРЕДНЯЯ
    *   **Описание**: Текущая архитектура "модульного монолита" может столкнуться с трудностями при независимом масштабировании отдельных компонентов, таких как импортеры или API, под высокой нагрузкой. Отсутствие явных стратегий масштабирования для импортеров и агентов, а также для базы данных (например, реплики для чтения), может стать узким местом. Неясно, является ли обслуживание HTML в `cmd/api` статическим или серверным рендерингом, что может повлиять на производительность.
    *   **Рекомендация**: Четко определить, как импортеры и агенты масштабируются (например, несколько экземпляров, использование очередей сообщений). Рассмотреть возможность использования реплик для чтения PostgreSQL, если нагрузка на чтение станет высокой. Уточнить, что `cmd/api` обслуживает статический HTML, а не выполняет серверный рендеринг.
*   **Отказоустойчивость**
    *   **Серьезность**: ВЫСОКАЯ
    *   **Описание**: В архитектуре отсутствует явное упоминание механизмов обработки ошибок, повторных попыток, идемпотентности или автоматических выключателей (circuit breakers) для критически важных компонентов, таких как импортеры, агенты и внешние интеграции (Telegram API, OpenAI API). Неясно, как система восстанавливается после сбоев импорта или задач агентов, и как обеспечивается целостность данных в таких сценариях.
    *   **Рекомендация**: Внедрить надежную обработку ошибок, механизмы повторных попыток с экспоненциальной задержкой и идемпотентность для всех операций импорта и задач агентов. Рассмотреть использование более устойчивой системы очередей для `agent_queue` (например, с персистентностью и гарантией доставки). Документировать режимы отказа и стратегии восстановления.
*   **Наблюдаемость**
    *   **Серьезность**: ВЫСОКАЯ
    *   **Описание**: В архитектуре полностью отсутствуют упоминания о стратегиях логирования, мониторинга, оповещения и трассировки. Это критический пробел для любой производственной системы, так как без этих инструментов будет крайне сложно отслеживать состояние системы, выявлять проблемы производительности или ошибки.
    *   **Рекомендация**: Внедрить комплексную стратегию наблюдаемости, включающую: структурированное логирование (например, с использованием Zap или Logrus), сбор метрик (например, Prometheus/Grafana), систему оповещений (например, Alertmanager) и распределенную трассировку (например, OpenTelemetry).
*   **Согласованность**
    *   **Серьезность**: СРЕДНЯЯ
    *   **Описание**: Хотя упоминается "normalized-first media and comment graph" и требование "Import and dedupe flows must preserve referential consistency", неясно, как именно эти гарантии согласованности обеспечиваются на практике (например, с помощью транзакций, конкретной логики дедупликации, обработки ошибок). "Legacy JSON payloads tolerated only as compatibility/fallback layer" представляет потенциальный риск несогласованности, если не будет четкого плана по его устареванию и удалению.
    *   **Рекомендация**: Детализировать механизмы обеспечения ссылочной целостности во время импорта (например, использование транзакций, строгие проверки данных, идемпотентные операции). Определить четкий путь устаревания и удаления устаревших JSON-полезных нагрузок.
*   **Поддерживаемость**
    *   **Серьезность**: СРЕДНЯЯ
    *   **Описание**: Разделение операционных команд между Windows и WSL ("The project spans Windows and WSL, so operational commands must be explicit about which shell owns them") может привести к путанице, ошибкам и увеличению накладных расходов на поддержку. Наличие "legacy JSON payloads" также добавляет технический долг и сложность.
    *   **Рекомендация**: Стандартизировать использование оболочки, по возможности, для всех операций, связанных с бэкендом и базой данных (например, все в WSL). Разработать четкую стратегию по устареванию и удалению устаревших JSON-полезных нагрузок.

### 2. Соответствие Конституции

Архитектура в целом соответствует принципам, изложенным в Конституции `MUDRO`.

*   **Стек**: Полное соответствие. Упомянуты все компоненты стека.
*   **Стандарты кодирования**: Не нарушает, хотя напрямую не описывает. Упоминание отсутствия секретов в репозитории соответствует.
*   **Тестирование**: Не нарушает. Упоминание безопасности тестов от воздействия на рабочие БД соответствует.
*   **Ограничения**: Полное соответствие. `Makefile`, WSL/Linux контур для операций, требование одобрения для рискованных действий с БД, политика источников Telegram/VK — все это отражено.
*   **Безопасность**: Полное соответствие. Отсутствие секретов в репозитории, доступ к БД только по loopback, требование одобрения для публичных API изменений, безопасность тестов — все это учтено.
*   **Правила LLM**: Не нарушает, так как это правила для взаимодействия с LLM, а не для самой архитектуры.

### 3. Спорные решения и альтернативы

*   **"Модульный монолит" против микросервисов**
    *   **Решение**: Текущее состояние — модульный монолит с планами на событийно-ориентированное разделение.
    *   **Спорность**: Прагматично на ранних этапах, но упоминание "планируемого" разделения и Kafka указывает на будущее направление к микросервисам. Слишком долгое сохранение монолита может затруднить независимое развертывание, масштабирование и автономию команд. Слишком ранний переход к микросервисам добавляет ненужную сложность.
    *   **Альтернатива**: Четко определить критерии для перехода к микросервисам (например, размер команды, конкретные узкие места масштабирования, сложность домена). Рассмотреть паттерн "Strangler Fig" для постепенной миграции, если микросервисы являются долгосрочной целью.
    *   **Серьезность**: СРЕДНЯЯ (стратегическое решение с долгосрочными последствиями).
*   **Разделение операций между Windows и WSL**
    *   **Решение**: "Проект охватывает Windows и WSL, поэтому операционные команды должны явно указывать, какая оболочка ими владеет."
    *   **Спорность**: Это увеличивает когнитивную нагрузку и потенциал для ошибок. Разработчикам приходится помнить, какую среду использовать для какой команды.
    *   **Альтернатива**: Стандартизировать WSL для *всех* операций разработки и эксплуатации, связанных с бэкендом и базой данных. Фронтенд может оставаться на Windows, но инструменты бэкенда должны быть согласованы. Это упрощает ментальную модель.
    *   **Серьезность**: СРЕДНЯЯ (опыт разработчиков и потенциал для ошибок).
*   **`tmp/` для вспомогательных скриптов**
    *   **Решение**: "`tmp/` может содержать вспомогательные скрипты и не должен определять канонический статус верификации."
    *   **Спорность**: Хотя это полезно для быстрых экспериментов, `tmp/` может стать свалкой для неподдерживаемого, недокументированного и потенциально некорректного кода. Это размывает границу между "черновиком" и "производственно-значимым помощником".
    *   **Альтернатива**: Для любых действительно полезных и потенциально переиспользуемых вспомогательных скриптов перенести их в соответствующую директорию `scripts/` или внутренний пакет `internal/` с документацией и тестами. Строго соблюдать правило, что `tmp/` является временным и не является частью кодовой базы.
    *   **Серьезность**: НИЗКАЯ (потенциал для проблем с гигиеной кода).
*   **Kafka как "опциональное будущее/основное направление"**
    *   **Решение**: Kafka упоминается как будущее направление, но не как обязательная зависимость во время выполнения.
    *   **Спорность**: Внедрение распределенной системы обмена сообщениями, такой как Kafka, является значительным архитектурным решением с высокими операционными накладными расходами. Упоминание ее как "опционального будущего" без четких триггеров или вариантов использования может привести к преждевременной оптимизации или неподходящему решению.
    *   **Альтернатива**: Вместо того чтобы называть конкретную технологию, описать *потребность* в асинхронной связи или потоковой передаче событий, когда она возникнет (например, "асинхронная обработка длительных импортов", "источники событий для аудита"). Оценивать Kafka (или другие альтернативы, такие как RabbitMQ, NATS, облачные очереди) только тогда, когда эта потребность станет конкретной.
    *   **Серьезность**: НИЗКАЯ (потенциал для преждевременного архитектурного обязательства).

### 4. Рекомендации по ADR

*   **ADR 001: Стратегия эволюции от монолита к микросервисам**
    *   **Контекст**: Текущая архитектура представляет собой модульный монолит с планами на событийно-ориентированное разделение и Kafka.
    *   **Решение**: Определить стратегию эволюции от модульного монолита к более распределенной архитектуре. Указать триггеры, критерии и потенциальные паттерны (например, Strangler Fig, ограниченные контексты).
    *   **Последствия**: Обеспечивает ясность в долгосрочном архитектурном видении и предотвращает преждевременное или нескоординированное разделение.
*   **ADR 002: Стратегия наблюдаемости**
    *   **Контекст**: В текущем проекте архитектуры отсутствует явное упоминание логирования, мониторинга, оповещения и трассировки.
    *   **Решение**: Определить выбранные инструменты и практики для наблюдаемости по всем компонентам (бэкенд, фронтенд, база данных, импортеры, агенты).
    *   **Последствия**: Гарантирует, что система может быть эффективно мониторирована, отлажена и поддерживаема в продакшене.
*   **ADR 003: Стратегия отказоустойчивости и устойчивости**
    *   **Контекст**: Отсутствие явных механизмов отказоустойчивости для импортеров, агентов и внешних интеграций.
    *   **Решение**: Описать общий подход к отказоустойчивости, включая обработку ошибок, политики повторных попыток, идемпотентность и автоматические выключатели для критически важных операций и внешних вызовов.
    *   **Последствия**: Повышает надежность и устойчивость системы, особенно для приема данных и фоновой обработки.
*   **ADR 004: Стандартизация кросс-ОС среды разработки**
    *   **Контекст**: Проект охватывает Windows и WSL, что приводит к потенциальной операционной сложности.
    *   **Решение**: Стандартизировать настройку среды разработки, явно определяя, какие операции выполняются в WSL, а какие в Windows, или предлагая унифицированный подход (например, все операции бэкенда/БД в WSL).
    *   **Последствия**: Уменьшает трения для разработчиков, улучшает согласованность и минимизирует проблемы, связанные со средой.

## Предлагаемая Архитектура

# Архитектура

## Обзор
`MUDRO` — это контент-платформа, построенная вокруг унифицированной ленты данных из источников Telegram и VK, с бэкендом на Go, хранилищем PostgreSQL, конвейерами импорта, фронтендом на React и операционными ботами. Текущая среда выполнения все еще близка к модульному монолиту, но кодовая база уже разделяет API, импортеры, потоки ботов/отчетов и контуры обслуживания, подобные агентам.

## Компоненты
- `cmd/api`
  - Обслуживает JSON-эндпоинты, такие как `/api/front`, `/api/posts`, `/healthz`, и статические HTML-страницы ленты.
- Импортеры
  - `cmd/vkimport`, `cmd/tgimport`, `cmd/tgload`, `cmd/tgcsvimport`, `cmd/tgcommentsimport`, `cmd/tgcommentscsvimport`, `cmd/tgcommentmediaimport`, `cmd/tgdedupe`, `cmd/tgrootmerge`.
  - Отвечают за безопасную нормализацию и загрузку исходных данных. Разработаны для идемпотентности и устойчивости к сбоям, с использованием транзакций для обеспечения согласованности данных. Масштабируются путем запуска нескольких экземпляров, при необходимости координируемых через очереди сообщений.
- Боты и отчетность
  - `cmd/bot` для управления Telegram и взаимодействия с проектом.
  - `cmd/reporter` для потоков дайджестов/отчетов.
- Контур агентов
  - `cmd/agent` плюс `internal/agent` для паттернов планировщика/работника и выполнения с контролем. Задачи агентов обрабатываются через `agent_queue`, которая поддерживает персистентность и механизмы повторных попыток для обеспечения отказоустойчивости.
- Фронтенд
  - `frontend/` приложение React + TypeScript, потребляющее живой API.
- Операции и развертывание
  - `Makefile`, `docs/ops-runbook.md`, `scripts/ops/*`, `docker-compose*.yml`.

## Хранение данных
- Основное реляционное хранилище: PostgreSQL.
- Основные таблицы:
  - `posts`
  - `post_comments`
  - `post_reactions`
  - `comment_reactions`
  - `media_assets`
  - `post_media_links`
  - `comment_media_links`
  - `agent_queue`
- Стратегия модели данных:
  - Сначала нормализованный граф медиа и комментариев.
  - Устаревшие JSON-полезные нагрузки допускаются только как слой совместимости/резерва, где это все еще необходимо, с четким планом по их устареванию и удалению.
- Локальный канонический DSN:
  - `postgres://postgres:postgres@localhost:5433/gallery?sslmode=disable`
- Масштабирование БД: При необходимости могут быть добавлены реплики для чтения для распределения нагрузки на чтение.

## Коммуникация
- Основной путь приложения: HTTP JSON от фронтенда к `cmd/api`.
- Локальный операционный путь: `Makefile` + оболочка WSL + `docker compose`.
- Боты используют Telegram API и внутреннее состояние проекта.
- Планируется событийно-ориентированное разделение; Kafka существует как опциональное будущее/основное направление, но не является обязательной зависимостью во время выполнения сегодня.

## Инфраструктура
- Локальная разработка:
  - репозиторий на `D:\mudr\mudro11`
  - путь WSL `/mnt/d/mudr/mudro11`
  - Docker Compose для БД
  - команды бэкенда из WSL (рекомендуется для всех операций бэкенда/БД)
  - команды фронтенда из оболочки Windows или WSL
- VPS:
  - `nginx` обслуживает фронтенд и обратное проксирование `/api`, `/media`, `/healthz`
  - сервисы бэкенда работают под управлением systemd / локального управления процессами
  - доступ к базе данных должен оставаться только по loopback и не быть публичным

## Внешние интеграции
- Telegram Bot API
- OpenAI API для функций и автоматизации на основе LLM
- Опциональные/публичные хостинговые слои, такие как Vercel, являются вспомогательными, а не канонической средой выполнения.
- Опциональные локальные/резервные среды выполнения LLM могут появиться позже, но не предполагаются в стандартной локальной настройке.
- Все внешние вызовы API реализуют механизмы повторных попыток с экспоненциальной задержкой и автоматические выключатели для повышения устойчивости.

## Безопасность
- Никаких деструктивных операций с БД без явного одобрения.
- Никаких секретных материалов в отслеживаемых артефактах репозитория.
- Изменения публичного API должны быть преднамеренными и рецензироваться.
- Локальные тесты и скрипты не должны неявно нацеливаться на производственные или общие БД.
- Потоки импорта и дедупликации должны сохранять ссылочную целостность для комментариев, медиа и реакций.

## Наблюдаемость
- **Логирование**: Структурированное логирование (например, с использованием Zap) для всех компонентов бэкенда, с уровнями логирования, настраиваемыми через переменные среды. Логи агрегируются и доступны для анализа.
- **Метрики**: Сбор метрик производительности и состояния (например, время отклика API, использование ресурсов, количество обработанных импортов) с использованием Prometheus-совместимых экспортеров.
- **Оповещения**: Настройка оповещений (например, через Alertmanager) на основе критических метрик и ошибок в логах для оперативного реагирования на проблемы.
- **Трассировка**: Внедрение распределенной трассировки (например, с использованием OpenTelemetry) для отслеживания запросов по всей системе, особенно через API и импортеры, для облегчения отладки и профилирования производительности.

## Известные компромиссы
- Проект охватывает Windows и WSL, поэтому операционные команды должны быть явными в отношении того, какая оболочка ими владеет. Рекомендуется использовать WSL для всех операций бэкенда и БД.
- `tmp/` может содержать вспомогательные скрипты и не должен определять канонический статус верификации. Любые полезные скрипты должны быть перенесены в `scripts/` или `internal/` с соответствующей документацией.
- Система еще не является полностью разделенной микросервисной платформой; некоторые документы описывают будущую структуру, в то время как среда выполнения остается намеренно прагматичной.
- VK сохраняется как контент только для снимков, в то время как активная работа по актуализации сосредоточена на Telegram.

## Архитектурные Инварианты Системы

### Ограничения потока данных
- Все мутации данных для основных сущностей (посты, комментарии, медиа) должны проходить через четко определенные эндпоинты API или идемпотентные процессы импорта/агентов.
- Прямые записи в базу данных вне `cmd/api` или утвержденной логики `cmd/importer`/`cmd/agent` запрещены.
- Все входящие данные из внешних источников должны проходить через этап нормализации и валидации перед сохранением в PostgreSQL.

### Контракты производительности
- P95 задержка для эндпоинтов `/api/front` и `/api/posts` должна быть ниже 200 мс при типичной нагрузке.
- Процессы импорта должны стремиться к пропускной способности не менее 100 элементов/секунду для Telegram-постов и комментариев.
- Время отклика `healthz` должно быть менее 50 мс.

### Инварианты безопасности
- Все внешние эндпоинты API (за исключением `/healthz`) требуют аутентификации и авторизации.
- Секреты должны загружаться из переменных среды или безопасного управления конфигурацией, никогда не должны быть жестко закодированы или зафиксированы в репозитории.
- Подключения к базе данных из сервисов приложений должны использовать роли с минимальными привилегиями.
- Все входящие запросы API должны проходить через валидацию входных данных для предотвращения распространенных уязвимостей (например, SQL-инъекций, XSS).

### Правила согласованности
- Все операции импорта и дедупликации должны быть идемпотентными и транзакционными для обеспечения целостности данных.
- Ограничения ссылочной целостности в PostgreSQL должны поддерживаться для таблиц `posts`, `post_comments`, `media_assets` и их связующих таблиц.
- Внешние вызовы API (Telegram, OpenAI) должны реализовывать механизмы повторных попыток с экспоненциальной задержкой и автоматические выключатели для обработки временных сбоев и предотвращения каскадных отказов.
- Устаревшие JSON-полезные нагрузки должны иметь четкий план миграции и удаления, чтобы избежать расхождений в данных.
