# ADR-007: Установление WSL + Docker Compose для локальной среды разработки

**Status:** proposed
**Date:** 2026-03-19

## Context
Проекту требовалась согласованная, воспроизводимая и изолированная локальная среда разработки, которая объединяет возможности хоста Windows с нативными инструментами Linux для бэкенда и операций с базой данных.

## Decision
Была выбрана комбинация Windows Subsystem for Linux (WSL) и Docker Compose. WSL предоставляет среду Linux для выполнения команд бэкенда Go, команд `make` и миграций базы данных, обеспечивая согласованность с серверной средой. Docker Compose изолирует базу данных PostgreSQL, предоставляя чистый, воспроизводимый экземпляр базы данных без загрязнения хост-системы. Команды фронтенда могут выполняться как из Windows, так и из WSL для гибкости.

## Alternatives
1.  **Чистая разработка в Windows (без WSL)**: Запуск всех инструментов нативно в Windows. Отклонен, поскольку это приводит к несоответствиям платформы (например, различные команды оболочки, пути к файлам, среды сборки Go) по сравнению с сервером на базе Linux, что затрудняет поддержку команд `Makefile` и скриптов и приводит к проблемам типа «работает на моей машине».
2.  **Полная виртуальная машина Linux (например, VirtualBox)**: Обеспечивает полную изоляцию и полностью нативную среду Linux. Отклонен из-за более высоких накладных расходов на ресурсы, более низкой производительности файловой системы для общих папок и менее бесшовной интеграции с настольной средой хоста Windows по сравнению с WSL2.

## Consequences
- Positive: Согласованная среда с производственной (подобная Linux), изолированная база данных, воспроизводимая настройка, хорошая производительность для операций бэкенда, бесшовная интеграция с IDE и инструментами Windows.
- Negative: Требует первоначальной настройки и понимания WSL; потенциальные проблемы с производительностью файловой системы при доступе к файлам Windows из WSL (хотя WSL2 это смягчает); управление двумя различными средами оболочки (Windows/WSL) может сбивать с толку.
- Risks: Проблемы с конфигурацией WSL могут блокировать разработку; потребление ресурсов Docker Desktop; потенциальные проблемы с путями между Windows и WSL, если не обрабатывать их осторожно.
