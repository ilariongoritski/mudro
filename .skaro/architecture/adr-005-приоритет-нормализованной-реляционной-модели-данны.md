# ADR-005: Приоритет нормализованной реляционной модели данных для основных сущностей

**Status:** proposed
**Date:** 2026-03-19

## Context
Основные данные (посты, комментарии, медиа, реакции) имеют сложные, взаимосвязанные отношения, которые требуют строгой согласованности и эффективного выполнения запросов по этим сущностям.

## Decision
Для основных сущностей была выбрана нормализованная реляционная модель данных. Это означает, что данные хранятся в отдельных таблицах с внешними ключами (`posts`, `post_comments`, `media_assets`, `post_media_links` и т.д.) для минимизации избыточности данных, обеспечения целостности данных и поддержки сложных запросов и объединений. Устаревшие JSON-полезные данные допускаются только в качестве слоя совместимости/резерва, а не в качестве основной стратегии хранения для новых данных.

## Alternatives
1.  **Документоориентированный подход (например, хранение постов с встроенными комментариями/медиа в JSONB)**: Предлагает гибкость схемы и потенциально меньшее количество объединений для простого извлечения. Отклонен, поскольку это приводит к дублированию данных (например, медиа-активы, на которые ссылаются в нескольких постах/комментариях), усложняет обновления (например, изменение медиа-актива требует обновления всех встроенных копий) и усложняет проверки ссылочной целостности, которые критически важны для обеспечения безопасности данных проекта.
2.  **Хранилище ключ-значение (например, Redis для основных данных)**: Чрезвычайно быстро для простых поисков. Отклонен, поскольку ему не хватает возможности определять сложные отношения, выполнять транзакционные операции по нескольким «ключам» или выполнять богатые запросы, необходимые для ленты контента и функций отчетности. Он непригоден для основного хранения взаимосвязанных сущностей домена.

## Consequences
- Positive: Высокая целостность данных, уменьшенная избыточность данных, эффективные сложные запросы через объединения, четкие отношения между сущностями, более простое обслуживание данных и проверки согласованности.
- Negative: Требует более сложных запросов (объединений) для извлечения агрегированных данных; изменения схемы могут быть более сложными (миграции); потенциальные проблемы с производительностью при чрезмерных объединениях на очень больших наборах данных, если они не проиндексированы должным образом.
- Risks: Чрезмерная нормализация может привести к «аду объединений», если не сбалансировать ее с соответствующей денормализацией для моделей чтения; требует тщательного планирования миграции для эволюции схемы.
