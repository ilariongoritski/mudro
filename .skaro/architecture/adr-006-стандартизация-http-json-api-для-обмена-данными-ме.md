# ADR-006: Стандартизация HTTP JSON API для обмена данными между фронтендом и бэкендом

**Status:** proposed
**Date:** 2026-03-19

## Context
Проекту требовался четкий, широко понятный и гибкий протокол связи между фронтендом на React и бэкендом на Go API.

## Decision
HTTP JSON API был выбран в качестве основного пути приложения. Это использует повсеместную природу HTTP, простоту и читаемость JSON, а также широкую поддержку в веб-браузерах и клиентских фреймворках. Это соответствует общим практикам веб-разработки и позволяет легко интегрироваться с различными клиентами.

## Alternatives
1.  **GraphQL**: Предлагает более эффективную выборку данных, позволяя клиентам запрашивать именно то, что им нужно, уменьшая избыточную выборку. Отклонен для начальной фазы из-за его дополнительной сложности в реализации (определение схемы, резолверы) и накладных расходов на внедрение нового языка запросов, что было признано ненужным для текущего объема API-взаимодействий.
2.  **gRPC**: Обеспечивает высокопроизводительную, строго типизированную связь с использованием Protocol Buffers, идеально подходящую для микросервисов и внутренней связи. Отклонен для связи между фронтендом и бэкендом из-за проблем совместимости с браузерами (требует прокси или специальных клиентских библиотек) и менее читаемой природы protobufs, что делает его менее подходящим для общедоступного веб-API.

## Consequences
- Positive: Простота, широкая поддержка браузеров и клиентов, человекочитаемый формат данных, легкая отладка с помощью стандартных инструментов браузера, хорошо подходит для RESTful моделирования ресурсов.
- Negative: Может привести к избыточной или недостаточной выборке данных, если не спроектирован тщательно; требует нескольких запросов для сложных графов данных; версионирование может быть сложным.
- Risks: Дизайн API может стать непоследовательным без строгих рекомендаций; потенциальные проблемы с производительностью при большом количестве небольших запросов или больших полезных данных.
