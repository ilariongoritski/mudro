# ADR-002: Использование Go 1.22+ для разработки бэкенда

**Status:** proposed
**Date:** 2026-03-19

## Context
Проекту требовался производительный, надежный и поддерживаемый язык бэкенда для API-сервисов, импортеров и фоновых задач, способный эффективно обрабатывать параллельные операции.

## Decision
Go 1.22+ был выбран за его сильную модель параллелизма (goroutines, channels), отличную производительность, статическую типизацию, быструю компиляцию, простое развертывание (единый бинарный файл) и растущую экосистему. Его явный поток данных и идиоматический стиль соответствуют стандартам кодирования проекта.

## Alternatives
1.  **Python (например, с FastAPI/Django)**: Предлагает быструю разработку и обширную экосистему, но обычно имеет более низкую производительность во время выполнения и более высокое потребление памяти по сравнению с Go, а также отсутствие сильной статической типизации и встроенных примитивов параллелизма Go, которые полезны для высокопроизводительных импортеров и API-сервисов. Отклонен из-за требований к производительности и параллелизму.
2.  **Node.js (TypeScript)**: Предоставляет единый язык для фронтенда и бэкенда, а также хорошую производительность ввода-вывода. Однако его однопоточный цикл событий (хотя и асинхронный) может быть менее эффективным для задач, связанных с интенсивными вычислениями, по сравнению с goroutines Go, а его экосистема может быть более фрагментированной. Отклонен из-за предпочтения явной модели параллелизма Go и гарантий времени компиляции для стабильности бэкенда.

## Consequences
- Positive: Высокая производительность, эффективное использование ресурсов, строгая типовая безопасность, простое развертывание, хорошая поддержка параллелизма, поддерживаемая кодовая база.
- Negative: Более крутая кривая обучения для разработчиков, незнакомых с идиомами Go; меньшая экосистема для некоторых специализированных библиотек по сравнению с Python/Node.js.
- Risks: Возможность сложной обработки ошибок, если не управлять ею тщательно; зависимость от конкретных версий Go для совместимости инструментария.
