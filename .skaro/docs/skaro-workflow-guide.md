# Подробный гайд по Skaro для MUDRO

## 1. Что такое Skaro простыми словами

`Skaro` в проекте `MUDRO` нужно понимать не как сайт для пользователей и не как
автономного программиста, а как внутренний проектный cockpit.

Его задача:

- держать в одном месте правила проекта;
- хранить архитектурную рамку;
- собирать milestones и задачи;
- показывать текущее состояние работы через dashboard;
- запускать проверки и LLM-фазы в управляемом виде.

Важная мысль:

- код, база, API, фронтенд и деплой живут в самом репозитории `mudro`;
- `Skaro` живет рядом с кодом и помогает этим управлять;
- `.codex/*` не заменяется и не выбрасывается.

## 2. Что именно Skaro дает в MUDRO

На вашем текущем сетапе `Skaro` уже полезен как слой порядка поверх живого
проекта.

Сейчас он закрывает такие задачи:

- дает локальный dashboard по адресу `http://127.0.0.1:4700/dashboard`;
- показывает pipeline проекта: `Конституция -> Архитектура -> ADR -> План -> Задачи -> Ревью`;
- держит milestones и task-карточки;
- хранит LLM-конфигурацию по ролям;
- умеет запускать проверки через `validate`;
- связывается с вашим текущим workflow `VS Code + .codex + repo`.

Что он не делает:

- не заменяет GitHub;
- не заменяет `VS Code`;
- не заменяет ваш backend/frontend runtime;
- не делает проект магически готовым без заполнения структуры.

## 3. Как Skaro встроен в ваш сетап

Для `MUDRO` сделана локальная привязка под Windows + WSL + `D:` диск.

Ключевые точки:

- launcher: `scripts/skaro-local.ps1`
- конфиг: `.skaro/config.yaml`
- секреты: `.skaro/secrets.yaml`
- guide и docs: `.skaro/docs/*`
- milestones и tasks: `.skaro/milestones/*`
- задачи `VS Code`: `.vscode/tasks.json`

Уже настроены роли:

- default: `OpenAI / gpt-5.4`
- architect: `OpenAI / gpt-5.4`
- coder: `OpenAI / gpt-5.3-codex`
- reviewer: `OpenAI / gpt-5.4`

Уже настроены verify-команды под ваш проект:

- `make dbcheck`
- `make tables`
- `go test` без `tmp`
- `frontend build`
- `frontend lint`

## 4. Что нужно перед первым использованием

Минимальный набор:

- локальный репозиторий `D:\mudr\mudro11`
- рабочий `Skaro` launcher `scripts/skaro-local.ps1`
- `OPENAI_API_KEY` в `.skaro/secrets.yaml`

Дополнительно полезно:

- держать проект открытым в `VS Code`;
- запускать `Skaro UI` отдельной задачей;
- не закрывать терминал, пока открыт dashboard.

## 5. Как запустить Skaro

### Через VS Code

Самый удобный сценарий:

1. Откройте проект `D:\mudr\mudro11` в `VS Code`.
2. Откройте `Terminal -> Run Task`.
3. Запустите `Skaro: UI`.
4. Откройте `Skaro: Open Dashboard` или вручную откройте:
   `http://127.0.0.1:4700/dashboard`

### Через PowerShell

Если без `VS Code`:

```powershell
cd D:\mudr\mudro11
powershell -ExecutionPolicy Bypass -File .\scripts\skaro-local.ps1 ui --port 4700 --no-browser
```

Пока это окно открыто, dashboard работает.

Если окно закрыть, `Skaro UI` остановится.

## 6. Какой способ работы оптимален

Для `MUDRO` оптимален такой режим:

- `VS Code` = место, где вы редактируете код и markdown-файлы;
- `Skaro UI` = место, где вы смотрите проектную картину и переходите по задачам;
- `CLI / tasks` = место, где вы запускаете статус и проверки;
- `.codex/*` = оперативная память проекта и run-логи;
- `.skaro/*` = структурированная проектная рамка.

Проще говоря:

- писать и править лучше в файлах;
- смотреть состояние и навигироваться удобно через dashboard.

## 7. Что находится внутри `.skaro`

### 7.1 Конституция

Файл:

- `.skaro/constitution.md`

Сюда кладутся только устойчивые правила проекта:

- стек;
- ограничения по безопасности;
- правила изменения API;
- базовые инженерные принципы;
- что допустимо, а что нет.

Не стоит класть сюда:

- разовые задачи;
- временные заметки;
- дневник сессии.

### 7.2 Архитектура

Файлы:

- `.skaro/architecture/architecture.md`
- `.skaro/architecture/invariants.md`

Сюда попадают стабильные факты:

- из каких частей состоит система;
- как связаны backend, DB, frontend и ops-контур;
- какие инварианты нельзя ломать;
- какие data-flow считаются каноническими.

### 7.3 ADR

Раздел:

- `.skaro/adr/*` или документированный каталог ADR, если вы расширите структуру

ADR нужны для решений, которые не должны теряться:

- почему media-нормализация сделана именно так;
- почему VK считается snapshot-only;
- почему frontend поднят self-hosted на VPS;
- почему `.codex` и `Skaro` живут параллельно.

### 7.4 План

Файл:

- `.skaro/devplan.md`

Это не список мелких задач, а верхнеуровневый план:

- основные контуры работы;
- последовательность крупных этапов;
- что уже есть;
- что блокирует следующий шаг.

### 7.5 Milestones и задачи

Основная рабочая зона:

- `.skaro/milestones/<milestone>/milestone.md`
- `.skaro/milestones/<milestone>/order.json`
- `.skaro/milestones/<milestone>/<task>/spec.md`
- `.skaro/milestones/<milestone>/<task>/plan.md`

Именно отсюда dashboard начинает быть полезным.

## 8. Как правильно заполнять Skaro

### 8.1 Правило верхнего уровня

Заполняйте `Skaro` так, чтобы через неделю можно было открыть dashboard и без
чтения всей истории понять:

- что это за проект;
- над чем сейчас идет работа;
- какие решения уже приняты;
- что делать следующим шагом.

### 8.2 Как писать `constitution.md`

Хорошая конституция отвечает на вопросы:

- какой стек канонический;
- где нельзя рисковать данными;
- что нельзя коммитить;
- когда нужно остановиться и спросить пользователя;
- что считать готовностью изменения.

Для `MUDRO` туда логично включать:

- `Go + PostgreSQL + React`;
- `WSL` как основной shell-контур для `make/docker/go`;
- запрет на опасные DDL/DB-действия без подтверждения;
- запрет на коммит секретов и generated outputs;
- обязательные базовые проверки.

### 8.3 Как писать архитектуру

Хорошая архитектура не повторяет весь README, а оставляет только несущую схему:

- сущности: `posts`, `comments`, `media`, `reactions`;
- основной runtime: `api`, `frontend`, `bot`, `reporter`;
- импортные контуры;
- VPS-контур;
- инварианты по данным.

### 8.4 Как писать ADR

Пишите ADR только для решений, которые хочется быстро вспомнить позже.

Примеры для `MUDRO`:

- `ADR: VK is snapshot-only`
- `ADR: media normalization uses additive tables`
- `ADR: self-hosted frontend is primary MVP runtime`
- `ADR: .codex remains detailed memory, Skaro remains project cockpit`

### 8.5 Как писать `devplan.md`

План должен показывать:

- текущую стадию проекта;
- большие блоки работ;
- их порядок;
- критерий завершения блока.

Не нужно превращать `devplan.md` в копию `todo`.

### 8.6 Как писать задачи

Каждая нормальная задача должна иметь минимум:

- `spec.md`
- `plan.md`

`spec.md` отвечает на вопрос:

- что именно делаем и зачем.

`plan.md` отвечает на вопрос:

- какими шагами это будет делаться;
- какие проверки обязательны;
- где риск;
- какой DoD.

Хорошая задача в `Skaro` должна быть понятна без чтения длинного чата.

## 9. Как использовать `.codex` вместе со Skaro

Это критически важно для вашего проекта.

Правильное разделение такое:

- `.codex/*` = подробная память, лог сессий, runtime-шум, оперативная история;
- `.skaro/*` = структура, решения, milestones, task-spec, план.

Что уже сделано:

- снимки `.codex` импортированы в `.skaro/docs/imported/`;
- summaries вынесены в `.skaro/docs/*`.

Как пользоваться дальше:

- сырые логи оставляйте в `.codex/logs/*`;
- ключевые выводы переносите в `Skaro` как summary или task/ADR;
- не превращайте `Skaro` в свалку raw-логов.

## 10. Как пользоваться dashboard

### 10.1 Дашборд

Главная страница нужна для быстрого ответа на вопросы:

- pipeline уже заполнен или еще пустой;
- есть ли milestones и задачи;
- какие роли и модели настроены;
- сколько задач сейчас заведено;
- какой проект вообще открыт.

На dashboard полезно смотреть в начале дня и перед фиксацией состояния.

### 10.2 Задачи

Раздел `Задачи` нужен как рабочая доска.

Отсюда удобно:

- смотреть все задачи;
- фильтровать по milestone;
- видеть прогресс по фазам;
- открывать конкретную карточку.

### 10.3 Карточка задачи

Внутри задачи нужно проверять:

- есть ли нормальная спецификация;
- есть ли план;
- какой следующий шаг пишет `Skaro`;
- где сейчас остановлен процесс.

### 10.4 Настройки

Раздел `Настройки` нужен для:

- названия и описания проекта;
- языка;
- темы;
- настройки LLM по умолчанию;
- ролей;
- навыков.

Лучше воспринимать это как точку конфигурации cockpit, а не как продуктовую
админку сайта.

## 11. Рекомендуемая структура milestones для MUDRO

Сейчас уже заведена разумная базовая структура:

- `01-project-cockpit`
- `02-public-mvp-release`
- `03-vps-runtime`

Логика такая:

- `project-cockpit` = процесс и управление проектом;
- `public-mvp-release` = внешняя поверхность и публичный результат;
- `vps-runtime` = устойчивость серверного контура.

Если появятся новые большие направления, лучше добавлять milestone только когда
это реально отдельный контур работ.

## 12. Ежедневный сценарий работы

Ниже самый практичный режим именно для вас.

### Утром или в начале новой сессии

1. Открыть репозиторий в `VS Code`.
2. Запустить `Skaro: UI`.
3. Открыть dashboard.
4. Посмотреть активные milestones и задачи.
5. Сверить `.codex/todo.md` и `Skaro`, чтобы не потерять контекст.

### Перед реализацией

1. Открыть нужную задачу в `Skaro`.
2. Убедиться, что `spec.md` описывает цель.
3. Убедиться, что `plan.md` описывает шаги и проверки.
4. Только после этого идти в код.

### После реализации

1. Прогнать релевантные проверки.
2. Запустить `Skaro: Status`.
3. Запустить `Skaro: Validate`.
4. Обновить `.codex/done.md` и `.codex/state.md`.
5. При необходимости обновить `plan.md`, milestone или summary docs.

## 14. Сколько токенов он тратит

Пока вы просто:

- открываете dashboard;
- читаете milestones;
- редактируете markdown-файлы;
- запускаете `status` и часть локальных проверок,

токены почти не расходуются.

Токены тратятся, когда реально идут LLM-запросы:

- уточнение задачи;
- планирование через модель;
- ревью через роль;
- генерация или анализ через роли.

Значит, `Skaro` можно держать как cockpit почти бесплатно, а LLM-фазы включать
только там, где они действительно нужны.

## 15. Типовые проблемы и быстрые решения

### Проблема: `ERR_CONNECTION_REFUSED`

Причина:

- `Skaro UI` не запущен.

Решение:

```powershell
cd D:\mudr\mudro11
powershell -ExecutionPolicy Bypass -File .\scripts\skaro-local.ps1 ui --port 4700 --no-browser
```

### Проблема: dashboard пустой

Причина:

- не заведены milestones или задачи;
- не заполнены `spec.md` и `plan.md`.

Решение:

- создать milestone;
- создать task;
- заполнить как минимум `spec.md` и `plan.md`.

### Проблема: LLM-фазы не работают

Причина:

- нет `OPENAI_API_KEY` или неверно настроен provider.

Решение:

- проверить `.skaro/secrets.yaml`;
- проверить `.skaro/config.yaml`;
- открыть `Настройки` и сверить роли/модели.

### Проблема: непонятно, куда писать информацию

Ориентир:

- правила -> `constitution.md`
- схема системы -> `architecture.md`
- долговременное решение -> ADR
- большой контур работ -> `devplan.md`
- конкретная работа -> task `spec.md` + `plan.md`
- сырая история и run-логи -> `.codex/*`
