# mudro — правила для работяги (R1 local) и CI (R2)

## 0) Назначение
Этот документ — обязательный регламент для любого агента/ассистента (“работяги”), который выполняет команды и меняет репозиторий **mudro**.
Цели:
1) Никакой потери данных без явного согласия человека.
2) Детерминированный “health loop” (можно повторить на любой машине).
3) Прозрачные логи: что делали, что сломалось, что починили.

---

## 1) Канонические факты проекта (НЕ ГАДАТЬ)
- Корень репы (R1): `~/projects/mudro`
- Postgres в Docker Compose, наружный порт: **5433** (внутри контейнера 5432)
- Канонический DSN:
  `postgres://postgres:postgres@localhost:5433/gallery?sslmode=disable`
- Миграции в папке: `migrations/`
- Код на Go, тесты по умолчанию: `go test ./...`

---

## 2) Health loop (как доказать, что работяга “живой”)
Запускать из корня репы:

1) `make up`
2) `docker compose ps`
3) Проверка БД:
   - предпочтительно: `make dbcheck`
   - fallback: `psql "$(DSN)" -X -v ON_ERROR_STOP=1 -c "select 1;"`
4) Миграции:
   - предпочтительно: `make migrate`
   - fallback: `psql "$(DSN)" -X -v ON_ERROR_STOP=1 -f migrations/001_init.sql`
     (если имя файла другое — спросить человека, не угадывать)
5) Список таблиц:
   - предпочтительно: `make tables`
   - fallback: `psql "$(DSN)" -X -c "\dt"`
6) Тесты:
   - предпочтительно: `make test`
   - fallback: `go test ./...`
7) Санити-запрос:
   `psql "$(DSN)" -X -c "select count(*) from posts;"`

Интерпретация:
- контейнер БД `healthy` + `select 1` проходит + тесты проходят + таблица `posts` существует ⇒ работяга живой
- `count = 0` — нормально (данные ещё не импортированы)

---

## 3) Политика ретраев (fail-fast)
На каждый шаг health loop:
- максимум 2 попытки (первая + один повтор)
- если не помогло: СТОП и вопрос человеку
В вопросе обязательно:
- точная команда
- точные строки ошибки
- путь к логам (`.codex/logs/<run>/index.md`)

---

## 4) Разрешённое самовосстановление (без подтверждения, НЕРАЗРУШИТЕЛЬНО)
Агент может без спроса:

### 4.1 Повтор/мягкий рестарт
- повторить упавшую команду 1 раз
- мягко перезапустить окружение (БЕЗ удаления volume):
  - `make down`
  - `make up`
  - `docker compose ps`

### 4.2 Если БД “ещё стартует”
- подождать до 10 секунд и повторить `dbcheck` один раз

### 4.3 Если в Makefile нет таргетов health loop
Если `make dbcheck/migrate/tables` отсутствуют — агент может ДОБАВИТЬ минимальные таргеты в `Makefile`,
которые вызывают `psql` с каноническим DSN. После — прогнать health loop заново.
(Это считается безопасным изменением “инфраструктуры запуска”.)

### 4.4 Comment-only фикс в миграциях
Если `.sql` падает ТОЛЬКО из-за мусорных “комментариев” типа `////`,
агент может заменить их на валидные комментарии Postgres (`--` или `/* ... */`),
СТРОГО без изменения DDL/логики. Если есть сомнения — СТОП и вопрос человеку.

---

## 5) Запрещено без подтверждения (потенциально разрушительно)
Нельзя без явного “ДА” от человека:
- drop/truncate/reset БД
- `docker compose down -v` / удаление volumes
- `rm -rf` и любые массовые удаления
- любые изменения миграций, которые меняют схему/DDL (кроме comment-only из 4.4)
- любые изменения публичного JSON API контракта

---

## 6) Доступ к Docker (частый затык)
Если ошибка про docker socket/permissions (`/var/run/docker.sock`):
агент может только продиагностировать:
- `id`
- `groups`
- `ls -l /var/run/docker.sock`
- `docker version`
После этого СТОП и вопрос человеку.

---

## 7) Логи (ОБЯЗАТЕЛЬНО, НА РУССКОМ)
### 7.1 `.codex/state.md` (краткий статус)
Формат записи:

- Дата/время:
- Что запускал:
- Что прошло:
- Что упало (ошибка 5–15 строк):
- Что починил (если было):
- Следующий шаг:

### 7.2 `.codex/logs/<run>/index.md` (подробный прогон)
Для каждого прогона создать папку:
`.codex/logs/YYYYMMDD-HHMM/`

И записать:
1) Окружение (pwd, git branch, docker/compose версии если важно)
2) Команды по порядку
3) Ключевой вывод (ошибки полностью, остальное можно резать)
4) Ретраи и решения (почему повторил/почему остановился)
5) Изменения в репе (diff кратко)

---

## 8) Стоп-сигналы (когда обязательно спрашивать)
- шаг упал второй раз
- нужен reset/удаление данных
- непонятно имя миграции или порядок применения
- правка не “comment-only”
- затрагивается контракт API

---

## 9) Стартовый промпт для работяги (вставить в Codex/агента)
Ты — R1 (локальный работяга) проекта mudro.
Работай только в ~/projects/mudro. Следуй AGENTS.md строго.
Прогони health loop. Все логи пиши на русском в .codex/state.md и .codex/logs/<run>/index.md.
Никаких разрушительных действий без явного подтверждения.
Если после 3 попыток не получилось — остановись и задай вопрос человеку с точными ошибками и пояснениями что не сложилось.
