# mudro — регламент локального агента и CI

## 0) Назначение
Этот документ задает обязательные правила для любого агента, который работает в репозитории `mudro` локально или готовит изменения к сохранению в `main`.

Цели:
1. Не терять данные и не ломать рабочий контур без явного подтверждения человека.
2. Делать проверяемые, минимальные и воспроизводимые изменения.
3. Поддерживать локальную работу в Windows + WSL без догадок и лишнего git-шума.

---

## 1) Язык и стиль работы

### 1.1 Язык
- Все объяснения, логи и комментарии для человека писать на русском.
- Английский использовать только для:
  - кода
  - терминальных команд
  - точных текстов ошибок
  - API/library names
  - путей к файлам
  - commit messages при необходимости

### 1.2 Инженерный стиль
- Предпочитать простые решения, минимальные diff и работающий код.
- Не переписывать рабочие части системы без явной причины.
- Не добавлять абстракции “на будущее”, если они не нужны задаче прямо сейчас.
- Если риск ошибки заметный, сначала уточнить у человека или остановиться по stop-сигналу.

## 2) Источники истины и канонические факты

### 2.1 Источники истины по приоритету
1. Текущая задача пользователя.
2. Этот `AGENTS.md`.
3. `Makefile` как канонический источник рабочих команд.

5. `docs/worker-autonomy.md` для автономного режима локального агента.
6. `README.md` для общего обзора.

При расхождении команд доверять сначала `Makefile`, потом runbook, а не угадывать.

### 2.2 Канонические факты проекта
- Репозиторий: `github.com/goritskimihail/mudro`
- Корень репы в WSL/Linux: `~/projects/mudro`

- Backend: Go
- Frontend: React + TypeScript

- Канонический DSN:
  `postgres://postgres:postgres@localhost:5433/gallery?sslmode=disable`

- Основной вход для команд: `Makefile`
- По умолчанию `Makefile` использует контейнерный `psql` через `docker compose exec -T db ...` (`USE_DOCKER_PSQL=1`)

  - `env/common.env`
  - `env/api.env`
  - `env/agent.env`
  - `env/bot.env`
  - `env/reporter.env`

### 2.3 Платформенные правила
- Для `make`, `bash`, `docker compose`, `go test`, миграций и shell-логики предпочитать WSL/Linux-контур.
- Windows PowerShell допустим для чтения файлов, git-проверок и точечных локальных команд, если это безопаснее.
- Не смешивать без необходимости Windows-пути и WSL-пути в одной команде.

## 3) Старт каждой сессии
Всегда начинать из корня актуальной локальной репы.

Обязательный чеклист:
1. Прочитать рабочую память проекта:
   - `.codex/todo.md`
   - `.codex/done.md`
   - `.codex/top10.md`
   - `.codex/memory.json`
   - `.codex/time_runtime.json`
   - `.codex/tg_control.jsonl`
2. Проверить текущее состояние дерева:
   - `git status --short`
3. Не откатывать чужие изменения без явной просьбы.
4. Определить тип задачи:
   - документация
   - локальный recovery
   - backend
   - frontend
   - импорт/миграции
   - агентский контур
5. Если задача требует рабочего локального контура, начать с `make health` или `make worker-loop`.

## 4) Инварианты безопасности

### 4.1 Никогда не коммитить
- `data/`
- `out/`
- `.env`
- `var/log/`
- generated artifacts из `output/`
- токены, ключи, секреты, приватные дампы

### 4.2 Запрещено без явного “ДА” от человека
- `drop`, `truncate`, `reset` БД
- `docker compose down -v`
- удаление volume
- `rm -rf` и массовые удаления
- любые DDL-изменения миграций, кроме comment-only фикса мусорных комментариев
- изменения публичного JSON API-контракта
- force push и любые потенциально разрушающие git-операции

### 4.3 Если задача неясна
Если изменение может заметно затронуть данные, схему, публичный контракт или production-поведение, остановиться и задать короткий вопрос человеку до правки.

## 5) Стандартные workflow

### 5.1 Быстрый health loop
Предпочтительный вариант:
- `make health`

Этот таргет должен последовательно выполнить:
- `make up`
- `make ps`
- `make dbcheck`
- `make migrate`
- `make tables`
- `make test`
- `make count-posts`

Критерий “локальный агент живой”:
- контейнер БД `healthy`
- `select 1` проходит
- таблица `posts` существует
- тесты проходят
- `count(posts)` выполняется

`count = 0` допустим, если данные еще не импортированы.

### 5.2 Пошаговый ручной health loop
Если нужен контроль по шагам:
- `make up`
- `docker compose ps`
- `make dbcheck`
- `make migrate`
- `make tables`
- `make test`
- `make count-posts`

Fallback без `Makefile` допустим только если таргеты отсутствуют или сломаны:
- `docker compose up -d`
- `docker compose ps`
- `psql "$(DSN)" -X -v ON_ERROR_STOP=1 -c "select 1;"`
- `psql "$(DSN)" -X -v ON_ERROR_STOP=1 -f migrations/001_init.sql`
- `psql "$(DSN)" -X -c "\dt"`
- `go test ./...`
- `psql "$(DSN)" -X -c "select count(*) from posts;"`

### 5.3 Расширенный recovery после долгого перерыва
Если локальная схема могла отстать:
- `make up`
- `make dbcheck`
- `make migrate-all`
- `make tables`
- `make test`
- `make count-posts`

Если нужен точечный контроль, применять по порядку:
- `make migrate`
- `make migrate-agent`
- `make migrate-comments`
- `make migrate-agent-review`
- `make migrate-agent-events`
- `make migrate-media`
- `make migrate-comment-model`

### 5.4 Агентский контур
Команды:
- однократное планирование: `make agent-plan-once`
- planner в цикле: `make agent-plan`
- worker в цикле: `make agent-work`
- approve risky-задачи: `make agent-approve TASK_ID=<id>`
- reject risky-задачи: `make agent-reject TASK_ID=<id> REASON='...'`

Safe task kinds, которые можно выполнять без отдельного подтверждения, если они не ведут к разрушительным действиям:
- `health_check` -> `make test`
- `db_check` -> `make dbcheck`
- `tables_check` -> `make tables`
- `count_posts` -> `make count-posts`

Перед запуском planner/worker сначала привести схему БД в актуальное состояние.

### 5.5 Боты и сервисные команды
- Telegram bot: `make bot-run`
- Reporter bot: `make report-run`
- Синхронизация памяти: `make memento`

Не запускать ботов без нужных env-переменных из `.env` и `env/*.env`.

### 5.6 Импорт и backfill
- CSV импорт Telegram-постов:
  `make tg-csv-import CSV=/abs/path/file.csv`
- CSV импорт Telegram-комментариев:
  `make tg-comments-csv-import CSV=/abs/path/file.csv`
- Импорт media для комментариев:
  `make tg-comment-media-import DIR=/abs/path/dir`
- Backfill media:
  `make media-backfill`
- Backfill комментариев:
  `make comment-backfill`

### 5.7 Frontend workflow
Если менялся `frontend/`:
- локальная разработка:
  - `cd frontend`
  - `npm.cmd install`
  - `npm.cmd run dev`
- обязательные проверки перед сохранением:
  - `npm.cmd run build`
  - `npm.cmd run lint` если задача затрагивает UI/TS/состояние

Если работа идет из WSL и там настроен Node, допустимы `npm install`, `npm run dev`, `npm run build`, `npm run lint`.

### 5.8 Backend workflow
Если менялся Go-код:
- минимальный baseline:
  - `go test ./...`
- если затронут конкретный пакет, сначала можно прогнать точечные тесты, но перед завершением задачи нужен хотя бы релевантный общий прогон

### 5.9 Подготовка к сохранению в `main`
Перед сохранением или выдачей результата как “готово к main”:
1. Проверить `git status --short`.
2. Убедиться, что в diff нет generated/sensitive файлов.
3. Прогнать релевантные проверки:
   - backend -> `go test ./...`
   - frontend -> `npm.cmd run build` и при необходимости `npm.cmd run lint`
   - миграции/БД -> `make dbcheck`, `make tables`, профильные тесты
4. Обновить `.codex/done.md` и при необходимости `.codex/top10.md`.
5. Обновить `.codex/state.md`.
6. Если задача была крупной или был отдельный прогон, записать `.codex/logs/<run>/index.md`.

## 6) Политика ретраев и самовосстановления

### 6.1 Fail-fast
На каждый шаг health/recovery loop:
- максимум 2 попытки: первая + один повтор
- если не помогло: СТОП и вопрос человеку

В вопросе обязательно указать:
- точную команду
- точные строки ошибки
- путь к логу `.codex/logs/<run>/index.md`

### 6.2 Разрешено без подтверждения
- повторить упавшую команду 1 раз
- мягко перезапустить окружение:
  - `make down`
  - `make up`
  - `docker compose ps`
- посмотреть хвост логов:
  - `make logs`
- подождать до 10 секунд и повторить `make dbcheck`, если БД еще стартует
- добавить недостающие безопасные `Makefile`-таргеты для `dbcheck/migrate/tables`, если их нет
- заменить мусорные SQL-комментарии типа `////` на валидные Postgres-комментарии, строго без изменения DDL/логики

### 6.3 Docker-permission блокер
Если ошибка связана с `/var/run/docker.sock`, разрешена только диагностика:
- `id`
- `groups`
- `ls -l /var/run/docker.sock`
- `docker version`

После этого остановиться и спросить человека.

## 7) Логи, память и формат записей

### 7.1 `.codex/state.md`
Краткий статус сессии вести на русском.

Формат записи:
- Дата/время:
- Что запускал:
- Что прошло:
- Что упало (ошибка 5-15 строк):
- Что починил (если было):
- Следующий шаг:

### 7.2 `.codex/logs/<run>/index.md`
Для отдельного прогона создавать папку:
- `.codex/logs/YYYYMMDD-HHMM/`

И писать:
1. Окружение
2. Команды по порядку
3. Ключевой вывод
4. Ретраи и решения
5. Изменения в репе

### 7.3 Рабочая память проекта
Обязательные файлы:
- `.codex/todo.md`
- `.codex/done.md`
- `.codex/top10.md`
- `.codex/memory.json`
- `.codex/time_runtime.json`
- `.codex/tg_control.jsonl`

Правила:
1. В начале сессии читать все шесть файлов.
2. Найденные новые улучшения или мелкие проблемы добавлять в `.codex/todo.md`, если они не решаются сразу.
3. После завершения задачи добавлять краткий измеримый результат в `.codex/done.md`.
4. Только действительно значимые изменения сущности проекта поднимать в `.codex/top10.md`.
5. `/health`, `/now` и итоговые ответы опирать на эту память, а не на догадки.

## 8) Кодировки, переводы строк и git-шум

### 8.1 Обязательное правило
Все текстовые файлы репозитория хранить в предсказуемом текстовом формате:
- кодировка: `UTF-8`
- переводы строк: `LF`

### 8.2 Практика
- Перед правкой проблемного файла проверять, не бит ли он по кодировке.
- Не сохранять Markdown/JSON в UTF-16, CP1251 или “ANSI”.
- Не вносить массовую косметическую нормализацию файлов без необходимости по задаче.

### 8.3 Для локальной работы
- Опираться на `.editorconfig` и `.gitattributes`, чтобы Windows + WSL давали одинаковый результат.

---

## 9) Stop-сигналы
Нужно обязательно остановиться и спросить человека, если:
- шаг упал второй раз
- нужен reset/удаление данных
- непонятно, какую миграцию применять и в каком порядке
- правка миграции уже не comment-only
- затрагивается публичный API-контракт
- изменения конфликтуют с чужими незавершенными правками в дереве

## 10) Стартовый промпт для нового локального агента
Ты — локальный агент проекта `mudro`.
Работай только в корне актуальной локальной репы.
Сначала прочитай `.codex/todo.md`, `.codex/done.md`, `.codex/top10.md`, `.codex/memory.json`, `.codex/time_runtime.json`, `.codex/tg_control.jsonl`.
Потом проверь `git status --short`.
Если задаче нужен рабочий контур, сначала прогони `make health` или пошаговый эквивалент.
Все объяснения и логи пиши на русском.
Не делай разрушительных действий без явного подтверждения человека.
Если после двух попыток шаг не проходит, остановись и покажи точную команду, точную ошибку и путь к логу.
