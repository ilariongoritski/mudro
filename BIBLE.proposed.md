# BIBLE.proposed.md

Статус: draft.

## Процесс цикла
1. Малый инкремент (до 25 файлов / 800 строк diff).
2. `make selftest` перед коммитом.
3. PR-only, без прямых пушей в `main`.
4. Риски и откат описывать в PR.

## Ежедневная работа (daily routine)
Каждый день Мудроборос выполняет фиксированный цикл:

1. **Новая ветка**: `agent/<yyyymmdd>-daily-<slug>`. Любые правки — только в ней, никогда в `main` и не в старые `agent/*` ветки.
2. **Снимок и review**: отчёт в `docs/audit/<yyyymmdd>-review.md`. Фиксирует риски и находки.
3. **Покрытие тестами**:
   - `go test -cover ./...` по всем пакетам, где есть тесты.
   - Для каждого нового/изменённого пакета в `services/*` и `tools/*` проверить покрытие. Если пакет без тестов — записать в отчёт как долг.
   - `make selftest` должен включать `./services/...` (иначе зелёный selftest ничего не гарантирует).
   - Фронтенд: `npm run test` (vitest) запускать вместе с lint.
4. **Мелкие моменты**: закрыть все недочёты, найденные в review, которые попадают под автономное разрешение (Mission §3) — форматирование, логи, мёртвый код, make clean, line-endings, мелкие рефакторинги. Всё одним коммитом или серией мелких.
5. **Два критичных**: выбрать максимум **два** самых критичных пункта из отчёта и исправить в этом же цикле. Больше — следующий день. Это защита от раздутых PR.
6. **Selftest зелёный** перед пушем. Красный selftest = откатить изменения в ветке, не пушить.
7. **Push в GitHub по возможности**: `git push -u origin agent/<yyyymmdd>-daily-<slug>`. Если пуш невозможен (нет доступа/сети) — зафиксировать причину в отчёте, работа не теряется, остаётся в локальной ветке до следующего цикла.
8. **PR**: открыть PR из agent-ветки в `main` с описанием (зачем, что, как проверить, риски, откат). Уведомить владельца.

Ограничения: ≤25 файлов / ≤800 строк diff за цикл. Если критичных больше двух — выносить в отдельные циклы.

## Безопасность
- Секреты вне репозитория.
- Запрещены опасные инфраструктурные изменения без одобрения владельца.

## Аудит
- Append-only журнал в `docs/audit/`.
- Каждый daily-цикл оставляет запись в `docs/audit/<yyyymmdd>-review.md`.
