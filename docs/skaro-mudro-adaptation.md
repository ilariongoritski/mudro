# Как адаптировать пайплайн Skaro под MUDRO

## Зачем
Статья про Skaro полезна для `mudro` не как призыв внедрить еще один инструмент, а как описание дисциплины разработки:

`конституция -> архитектура -> ADR -> план -> задачи -> ревью`

У `mudro` большая часть этого контура уже есть. Рациональная адаптация для проекта — не строить отдельный оркестратор поверх репозитория, а использовать существующие артефакты как единый spec-driven workflow.

## Карта соответствия
- Конституция -> `AGENTS.md`, `.editorconfig`, `.gitattributes`, `env/*.env.example`
- Архитектура -> `README.md`, `docs/microservices-architecture.md`, `docs/ops-runbook.md`, `docs/worker-autonomy.md`, `docs/mcp-baseline.md`
- ADR -> `docs/adr/`
- План разработки -> `.codex/todo.md`, `.codex/top10.md`, крупные заметки в `.codex/notes/`
- Задачи -> `.codex/templates/task-spec.md`, `.codex/logs/<run>/index.md`, рабочий diff в репозитории
- Ревью -> `go test ./...`, `make health`, `make dbcheck`, `make tables`, frontend build/lint, browser smoke
- История и память -> `.codex/done.md`, `.codex/state.md`, `.codex/memory.json`, `.codex/time_runtime.json`, `.codex/tg_control.jsonl`

## 1. Конституция
В статье конституция — это документ с правилами для LLM и команды. В `mudro` эту роль уже выполняет `AGENTS.md`.

В текущем проекте в конституции уже зафиксированы:
- язык общения
- обязательное чтение `.codex/*` в начале сессии
- приоритет `Makefile` и runbook как источников истины
- запрет на разрушительные действия без подтверждения
- правила для БД, миграций, git, кодировок и Windows + WSL

Практический вывод: `AGENTS.md` уже является рабочей конституцией проекта, его не нужно дублировать отдельным AI-документом.

## 2. Архитектура
В Skaro архитектура хранится как единый утвержденный документ. В `mudro` разумнее держать архитектуру как набор стабильных документов по зонам ответственности:

- `README.md` — общий overview, запуск, команды, состав системы
- `docs/microservices-architecture.md` — целевая декомпозиция runtime и event-пайплайна
- `docs/ops-runbook.md` — recovery, VPS-контур, hardening, эксплуатационные сценарии
- `docs/worker-autonomy.md` — поведение локального агентского цикла
- `docs/mcp-baseline.md` — LLM/MCP-инфраструктура и границы доступа

Практический вывод: для `mudro` архитектура уже существует, но в split-формате. Это нормально для проекта на стадии активной сборки MVP.

## 3. ADR
Статья правильно подчеркивает ценность ADR: ключевые решения надо не только принимать, но и фиксировать.

Для `mudro` ADR нужны не на каждый шаг, а только на решения, которые меняют правила игры:
- модель данных `posts/comments/media/reactions`
- импортные пайплайны и канонический source-of-truth
- публичный API-контракт
- VPS/runtime hardening
- self-hosted frontend/reverse proxy
- политика источников вроде `VK snapshot-only`

Минимальный регламент для `mudro`:
- хранить ADR в `docs/adr/`
- создавать ADR только для значимых решений
- не заменять ADR обычными записями в `.codex/done.md`

Шаблон и правила лежат в `docs/adr/README.md`.

## 4. План разработки
В Skaro план хранится по milestones. В `mudro` уже есть практический эквивалент:

- `.codex/todo.md` — операционный backlog с приоритетами `P0/P1/P2/P3`
- `.codex/top10.md` — сжатое стратегическое резюме проекта
- `.codex/done.md` — измеримая история поставленных результатов
- `.codex/notes/` — черновики и отдельные направления

Это уже дает рабочий контур планирования без отдельного dashboard.

Рекомендуемый принцип:
- в `.codex/todo.md` держать только исполнимые задачи
- в `.codex/top10.md` держать только изменения уровня проекта
- в `.codex/done.md` писать только измеримый результат и эффект

## 5. Задачи
В статье каждая задача проходит цикл:

`спецификация -> уточнение -> план -> реализация -> тесты`

Именно этот кусок полезно перенести в `mudro` наиболее явно.

Для средних и крупных задач рекомендуется использовать `.codex/templates/task-spec.md`.

Минимальный рабочий поток для `mudro`:
1. Сформулировать цель и контекст задачи.
2. Явно записать, что входит в scope, а что нет.
3. Зафиксировать ограничения и риски.
4. До кодинга определить команды верификации.
5. Сделать минимальный diff.
6. Записать результат в `.codex/state.md` и `.codex/done.md`.
7. Если решение архитектурное — добавить ADR.

Для одноразовых мелких правок отдельный task-spec не обязателен; для задач, которые затрагивают несколько подсистем, БД, импорт или frontend+backend вместе, он уже окупается.

## 6. Ревью
В `mudro` ревью должно быть repo-native, а не "поговорили с моделью и поверили".

Базовые проверки по типу задачи:
- backend -> `go test ./...`
- БД и миграции -> `make dbcheck`, `make migrate`, `make tables`
- полный локальный контур -> `make health`
- frontend -> `cd frontend && npm.cmd run build && npm.cmd run lint`
- UI/MVP -> browser smoke, screenshot, проверка against live API

Дополнительные вопросы ревью:
- не нарушены ли запреты из `AGENTS.md`
- не появился ли риск для живой БД
- не размыт ли публичный API-контракт
- не потерян ли источник истины по данным
- добавлены ли тесты или хотя бы релевантная верификация

## 7. Что не нужно копировать из статьи буквально
Для `mudro` не стоит автоматически тянуть все механики Skaro:

- отдельный UI/dashboard пока не нужен
- автогенерация архитектуры из кода не критична
- не нужно заводить отдельную директорию на каждую микрозадачу
- нельзя прятать risky DB-операции за красивым AI-пайплайном

Проект сейчас выигрывает от простоты и прозрачности сильнее, чем от дополнительного слоя orchestration.

## 8. Минимальная целевая схема для MUDRO
Если сжать адаптацию до одного практического контура, то для `mudro` она выглядит так:

1. `AGENTS.md` задает конституцию.
2. `README.md` и `docs/*` задают архитектуру и эксплуатационный контекст.
3. `.codex/todo.md` задает операционный план.
4. Для нетривиальной задачи создается spec по шаблону.
5. Делается минимальный diff и прогоняются проверки из `Makefile`.
6. Результат фиксируется в `.codex/state.md` и `.codex/done.md`.
7. Для решений уровня проекта пишется ADR.

## 9. Итог
Адаптация статьи для `mudro` не требует отдельного продукта наподобие Skaro.

Достаточно дисциплинированно использовать уже существующий контур:

`AGENTS.md -> docs/* -> .codex/todo.md -> task-spec -> code diff -> tests -> .codex/state.md/.codex/done.md`

Это даст основную ценность статьи:
- контекст не теряется между сессиями
- решения становятся проверяемыми
- работа LLM опирается на явные артефакты, а не на импровизацию
- проект развивается без лишнего платформенного слоя
