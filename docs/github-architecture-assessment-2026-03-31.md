# Оценка текущего состояния GitHub-архитектуры Mudro (2026-03-31)

## Короткий вывод

Репозиторий выглядит как **сильная переходная архитектура**: база уже microservices-first, но фактический runtime пока гибридный (часть ответственности всё ещё сосредоточена в `feed-api`).

Если считать по шкале зрелости, текущее состояние — **6.5/10**:
- **сильная основа** (структура `services/*`, `tools/*`, `contracts/*`, активные compose-контуры);
- **незавершённая декомпозиция** (gateway/bff/auth/orchestration пока additive-слой, а не основной production path);
- **высокий потенциал** на ближайшие 1–2 спринта без тяжелого rewrite.

---

## Что уже хорошо (архитектурно)

1. **Явный microservices-first в документации и layout**
   - Зафиксированы активные сервисы и отдельные зоны репозитория (`services`, `tools`, `contracts`, `ops`, `frontend`).
   - Есть разделение на active/transitional/legacy области, что снижает риск хаотичных переносов.

2. **Канонический runtime-контур уже формализован**
   - Core-compose поднимает `db + redis + kafka + api + agent`.
   - Additive-контур отдельно подключает `auth-api`, `orchestration-api`, `api-gateway`.
   - Это хороший признак зрелой миграции, где можно вводить новые границы постепенно.

3. **Данные и домен уже достаточно богаты для социальной ленты**
   - В roadmap и памяти отражены посты, комментарии, реакции, media-нормализация и agent queue.
   - Это значит, что архитектурный фундамент строится вокруг реального продуктового графа данных, а не демо-схемы.

4. **Операционный слой уже не “самописный хаос”**
   - Есть runbook/compose-профили, health endpoints, проверочные make-таргеты.
   - Для текущего размера проекта это критично: меньше bus factor и меньше ручного tribal knowledge.

---

## Главные архитектурные риски сейчас

1. **Гибридность runtime-пути**
   - Несмотря на наличие gateway/auth/bff/orchestration, основной поток по-прежнему тянется через `feed-api`.
   - Риск: рост feature pressure может снова “утяжелить” один сервис и затормозить декомпозицию.

2. **Транзитный слой может стать постоянным**
   - В репо явно отмечены transitional зоны и compatibility-слой.
   - Если не задать критерии “definition of done” для выноса доменов из `feed-api`, переход может зависнуть на месяцы.

3. **Инфра готова частично, но без полного production hardening**
   - По roadmap ещё открыты HTTPS/MinIO/часть CI и mobile/admin-фронта.
   - Это не блокер разработки, но блокер стабильного масштабирования публичного продукта.

4. **Есть конфликт контекстов документации**
   - Часть документов описывает уже новую сервисную модель, часть — исторический baseline с `cmd/*`-центром.
   - Риск: новый контрибьютор может запускать не тот контур и получать ложные выводы о “неработающей архитектуре”.

## Оценка по ключевым архитектурным осям

- **Модульность сервисов**: 7/10
- **Чёткость границ доменов**: 6/10
- **Операционная зрелость**: 6/10
- **Наблюдаемость и события**: 6/10

## Что делать дальше (приоритетно)

### P0 (в ближайшие 1–2 недели)

1. **Зафиксировать целевой production path**
   - Явно выбрать, через что идет пользовательский трафик: `api-gateway` как главный edge или временно `feed-api`.
   - Добавить в README и runbook одну каноническую схему “request flow”.

2. **Определить границы выноса из feed-api**
   - Минимум: auth/admin/orchestration должны иметь ownership и контрактные тесты.
   - Ввести критерий “feature в новый домен не добавляется в feed-api”.

3. **Свести “архитектуру на бумаге” и “архитектуру в compose”**
   - Для каждого сервиса из catalog пометить: `active`, `pilot`, `legacy-compatible`.
   - Это резко снизит риск ошибочных запусков в новых сессиях.

### P1 (2–6 недель)

4. **Закрыть обязательный ops/security минимум**
   - HTTPS + секреты + MinIO + CI baseline.
   - Без этого сложно уверенно масштабировать пользовательский трафик и медиа.

5. **Подготовить realtime-контур как отдельную capability**
   - WS/SSE слой + Kafka consumer + фронтовая подписка.
   - Это даст архитектурный мост от “ленты” к “мессенджерному поведению”.

6. **Ускорить архитектурную обратную связь метриками**
   - Ввести простые архитектурные KPI: p95 latency по feed/auth, ошибка импорта, глубина очереди агента, MTTR по деплою.

---

## Резюме для решения “куда двигаться”

Состояние GitHub-архитектуры сейчас **хорошее для переходной фазы**: команда не в хаосе, есть каркас, есть roadmap, и уже сделаны болезненные инфраструктурные шаги.

Критическая задача не “переписать всё”, а **дожать переход**:
1) закрепить единый production path,
2) довести декомпозицию границ,
3) закрыть ops/security базу.

Если это сделать, Mudro сможет эволюционировать из “ленты с расширениями” в полноценный backend соцсети/мессенджера без архитектурного долга, который обычно ломает темп на втором этапе роста.
